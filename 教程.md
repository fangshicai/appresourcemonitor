# App Resource Monitor 开发教程

## 1. 当前项目状态

当前仓库是 Flutter 默认模板项目，主要业务入口是 `lib/main.dart`，还没有资源监控页面、MVVM 分层、GetX 路由或原生平台能力。

本阶段只完成开发规则和方案文档，不生成 Flutter 业务代码，不修改 Android/iOS 原生工程能力。

## 2. Skills 安装

官方推荐安装命令：

```bash
npx skills add flutter/skills --skill '*' --agent universal
npx skills add dart-lang/skills --skill '*' --agent universal
```

更新命令：

```bash
npx skills update
```

如果本地没有 `npx`，可使用等价安装方式：

```bash
git clone --depth 1 https://github.com/flutter/skills.git /tmp/flutter-skills
git clone --depth 1 https://github.com/dart-lang/skills.git /tmp/dart-skills
mkdir -p .agents/skills
cp -R /tmp/flutter-skills/skills/. .agents/skills/
cp -R /tmp/dart-skills/skills/. .agents/skills/
```

安装后，`.agents/skills` 应包含 `flutter-*` 和 `dart-*` 技能目录。重启 Codex 后，新安装的 skills 才会被当前工具环境完整识别。

## 3. 推荐目录结构

后续进入代码实现阶段时，建议按下面结构组织：

```text
lib/
  app/
    app.dart
    bindings/
    routes/
  modules/
    dashboard/
      views/
      view_models/
      bindings/
    app_detail/
      views/
      view_models/
      bindings/
  models/
    app_resource_snapshot.dart
    monitored_app.dart
  services/
    app_resource_monitor_service.dart
    app_action_service.dart
  platform/
    platform_resource_bridge.dart
    method_channel_resource_bridge.dart
```

目录职责：

- `app`：应用入口、全局路由、全局绑定和主题。
- `modules`：按页面或功能模块组织 View、ViewModel 和 Binding。
- `models`：跨层传递的数据模型。
- `services`：业务能力接口和用例编排。
- `platform`：Flutter 与原生平台之间的桥接层。

## 4. MVVM + GetX 数据流

推荐数据流：

```text
View -> ViewModel/Controller -> Repository/Service -> Native Bridge -> Root/Jailbreak Provider
```

实现原则：

- View 只展示状态和发送用户意图。
- ViewModel/Controller 使用 GetX 管理响应式状态。
- Service 统一提供资源监控和应用动作接口。
- Platform Bridge 封装 MethodChannel/EventChannel，不暴露原生命令细节。
- Android/iOS 原生层分别实现平台采集，Flutter 层只消费统一模型。

## 5. 权限模型

这个项目的目标能力超过普通应用权限范围，因此按 Root/Jailbreak 研究版设计。

Android 研究方向：

- 使用 `/proc` 获取进程信息和 CPU/内存指标。
- 使用 `dumpsys` 辅助读取内存、包和运行状态。
- 使用 `du` 统计应用数据或缓存目录。
- 使用 `am force-stop` 关闭后台进程。
- 使用 `pm uninstall` 或系统卸载 Intent 提供卸载入口。

iOS 研究方向：

- 普通 iOS App 不能枚举、监控、关闭或卸载其他 App。
- Jailbreak 环境可探索系统进程列表、应用容器目录和 SpringBoard/launchd 相关能力。
- iOS 能力必须单独标注研究环境、风险和兼容性限制。

## 6. 后续实施顺序

1. 搭建 Flutter 分层骨架：GetX 路由、Binding、ViewModel、Service 抽象和平台桥接接口。
2. 使用假数据实现应用列表和资源监控 UI，先验证交互和状态流。
3. 实现 Android Root Provider，读取应用列表、运行状态、CPU、内存和磁盘占用。
4. 接入 Android 关闭后台和卸载动作，并为无 Root 环境提供明确失败状态。
5. 研究 iOS Jailbreak Provider，先验证能否稳定获取进程和 Bundle 信息。
6. 为各平台能力增加权限检测、错误状态、用户确认和危险操作提示。
7. 补充 Widget Test、Service Unit Test 和平台桥接 Mock Test。

## 7. 验收标准

- `.agents/skills` 中存在 Flutter 和 Dart skills。
- `AGENTS.md` 明确项目目标、技术栈、架构规则、权限边界和禁止事项。
- `教程.md` 明确安装方式、目录规划、MVVM + GetX 数据流、平台限制和后续实施顺序。
- 本阶段不修改 `lib/`、`android/`、`ios/` 的业务或平台实现代码。
- 后续实现任何高权限能力前，必须先设计权限检测和失败提示。
